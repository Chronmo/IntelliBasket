# 编码规范与接口标准

> 适用项目：IntelliBasket / 客群智篮  
> 生效日期：2026-07-21

## 1. 基本原则

- 一个模块只承担一类职责，避免前端、接口、算法和 SQL 口径互相嵌套。
- 所有配置通过环境变量或配置文件注入，不在代码中硬编码密码、地址和端口。
- 数据处理必须可重复执行；同一批次重复运行不能产生重复业务记录。
- 关键指标必须有统一定义、来源表和验证查询。
- 公开函数、接口、SQL 表和核心算法必须提供简洁说明。
- 每个里程碑先验证再提交，提交中不得包含原始数据、密钥、缓存或构建产物。

## 2. 命名规范

### 2.1 通用代码

| 对象 | 规范 | 示例 |
| --- | --- | --- |
| 变量、函数、方法 | lowerCamelCase | `customerId`、`calculateRfmScore()` |
| 类、类型、组件 | PascalCase | `RfmService`、`BasketNetworkChart` |
| 常量 | UPPER_SNAKE_CASE | `DEFAULT_MIN_SUPPORT` |
| 布尔值 | `is`、`has`、`can`、`should` 开头 | `isCancelled`、`hasCustomerId` |
| 集合 | 使用复数名词 | `customers`、`associationRules` |
| 私有成员 | 单下划线前缀 | `_databaseSession` |

Python 文件名和包名保留生态通用的 `snake_case`，数据库对象也使用 `snake_case`；这是为了兼容 Python 导入规范及 Hive/MySQL 大小写行为。代码内部自定义标识和 JSON 字段统一使用驼峰命名。

### 2.2 数据库

- 库、表、分区和列名使用 `snake_case`。
- 表名以分层前缀开头：`ods_`、`dwd_`、`dws_`、`ads_`、`dim_`。
- 主键使用 `{entity}_id`，时间使用 `_date` 或 `_ts`，布尔标志使用 `is_`。
- 金额字段必须标明业务含义，如 `sales_amount`、`refund_amount`。
- Hive 表结构、MySQL 服务表和 API JSON 在映射层完成命名转换。

### 2.3 REST API

- 路径使用小写复数名词和 kebab-case，例如 `/api/v1/rfm-segments`。
- JSON 请求和响应字段使用 lowerCamelCase。
- 查询参数使用 lowerCamelCase，例如 `snapshotDate`、`minSupport`。
- 版本放在 URL 中，当前版本统一为 `/api/v1`。

## 3. API 标准

### 3.1 HTTP 方法

| 方法 | 用途 |
| --- | --- |
| GET | 查询资源，不修改服务端状态 |
| POST | 创建资源或执行具有复杂请求体的分析 |
| PUT | 完整替换资源 |
| PATCH | 局部更新资源 |
| DELETE | 删除资源 |

分析看板以 GET 为主；需要多条件组合的营销建议可以使用 POST，但不能用 POST 替代普通查询。

### 3.2 统一响应

成功响应：

```json
{
  "success": true,
  "data": {},
  "meta": {
    "requestId": "01J...",
    "generatedAt": "2026-07-21T09:30:00+08:00"
  }
}
```

失败响应遵循 Problem Details 思路：

```json
{
  "success": false,
  "error": {
    "code": "INVALID_QUERY_PARAMETER",
    "message": "minSupport must be between 0 and 1",
    "details": []
  },
  "meta": {
    "requestId": "01J..."
  }
}
```

### 3.3 状态码

- `200`：查询或分析成功。
- `201`：资源创建成功。
- `204`：删除成功且无响应体。
- `400`：参数格式或业务规则错误。
- `404`：资源不存在。
- `409`：重复或状态冲突。
- `422`：请求结构验证失败。
- `500`：未预期服务错误。
- `503`：数据库或分析依赖暂不可用。

### 3.4 分页、排序与筛选

- 页码：`page`，从 1 开始。
- 每页：`pageSize`，默认 20，最大 100。
- 排序：`sortBy` 与 `sortOrder=asc|desc`。
- 时间使用 ISO 8601；日期为 `YYYY-MM-DD`，时间带时区。
- 比率使用 0 到 1 的小数，展示层再转换为百分比。
- 金额返回数值与货币代码，不返回带符号字符串。

## 4. Python规范

- 启用类型注解，公共函数必须标注输入与返回类型。
- 数据模型使用 `dataclass` 或 Pydantic，不在模块间传递含义不明的字典。
- 文件、数据库连接和会话使用上下文管理器。
- 日志使用 `logging`，禁止用 `print` 记录服务运行状态；命令行结果输出除外。
- 捕获具体异常，禁止空 `except` 或无条件吞掉错误。
- 算法阈值集中在配置对象中，测试中可覆盖。

## 5. TypeScript与Vue规范

- 启用 TypeScript 严格模式。
- Vue 组件使用 `<script setup lang="ts">`。
- 组件名采用 PascalCase，组合式函数使用 `use` 前缀。
- API 类型集中维护，不在页面重复定义接口。
- 图表组件只负责展示，筛选、取数和业务转换由页面或 store 负责。
- 颜色、间距和字体使用 CSS 变量，避免散落魔法值。
- 所有交互控件提供标签、键盘焦点和空状态。

## 6. SQL规范

- 关键字大写，字段逐行对齐，复杂逻辑拆分为 CTE。
- 禁止 `SELECT *` 进入生产层表。
- 每张表声明业务粒度、分区字段、存储格式和字段注释。
- 金额统一使用 `DECIMAL`，禁止用浮点类型作为最终金额字段。
- 分母使用 `NULLIF` 或条件判断处理零值。
- ODS 保留原值，DWD 负责标准化，DWS 负责可复用汇总，ADS 面向具体页面。

## 7. Git管理规范

### 7.1 分支

- `main`：可运行、可演示的稳定版本。
- `agent/phase2-implementation`：第二阶段集成开发分支。
- 如需要独立修复，使用 `fix/{description}`。

### 7.2 Conventional Commits

提交格式：

```text
<type>(<scope>): <summary>
```

常用类型：

- `feat`：新增功能。
- `fix`：修复缺陷。
- `docs`：文档。
- `test`：测试。
- `refactor`：不改变行为的重构。
- `build`：依赖与构建。
- `chore`：工程维护。

本阶段计划提交：

1. `docs: establish project architecture and engineering standards`
2. `feat(data): add retail ingestion and warehouse layers`
3. `feat(analytics): implement dynamic rfm and basket mining`
4. `feat(api): add analytics service endpoints`
5. `feat(web): build intellibasket dashboard`
6. `test: add end-to-end validation and runbooks`

### 7.3 提交前检查

- 查看 `git status` 和当前 diff。
- 运行当前里程碑相关测试或静态检查。
- 明确暂存文件，避免把工作区其他项目带入仓库。
- 确认无密码、令牌、原始数据、缓存和大体积生成物。
- 提交后推送远端，保留阶段性可追溯记录。

## 8. 接口安全与可观察性

- 所有输入由 Pydantic 校验，SQL 查询使用参数绑定。
- CORS 只允许配置中的前端地址。
- 每个请求生成 `requestId` 并写入日志。
- 健康检查拆分为存活状态与依赖状态。
- 异常响应不暴露连接串、SQL、文件路径和调用栈。
- 演示环境不提供写入、删除客户数据的外部接口。
