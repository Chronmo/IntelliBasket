# 客群智篮：项目背景与技术栈总结

> 完整名称：客群智篮——基于 Hive 数仓、动态 RFM 分层与分群购物篮挖掘的零售精准营销决策系统  
> 对应选题：数据仓库模块 3——用户订单行为多维分析系统（RFM 模型 + 购物篮分析）  
> 项目阶段：第一阶段  
> 文档日期：2026-07-21

## 1. 项目摘要

零售企业积累了大量订单明细，但“有数据”并不等于“能决策”。传统经营报表通常只回答销售额、订单量和畅销商品等总体问题，容易忽略客户价值差异，也无法说明哪些商品在同一张订单中具有真实的关联关系。

本项目以 UCI Online Retail II 真实交易数据为基础，建设 `ODS → DWD → DWS → ADS` Hive 分层数据仓库，在可靠的发票号边界上形成客户、订单、商品和时间主题数据；随后使用动态 RFM 模型识别客户价值与状态变化，再在各 RFM 客群内部使用 FP-Growth 挖掘关联规则，最终通过 Web 系统呈现经营概览、客户分层、客群迁移、商品组合网络和组合营销建议。

项目核心思想是：

> 先判断“应该服务谁”，再判断“向这类客户组合推荐什么”。

## 2. 项目背景

### 2.1 业务背景

电子商务与零售业务每天都会产生订单、商品、客户、时间和地域等多维交易数据。企业常见的经营问题包括：

- 所有客户使用同一套促销方案，营销资源利用率低。
- 高价值客户、潜力客户和流失风险客户缺少统一、可解释的识别标准。
- 畅销商品统计只能反映单品热度，不能发现真实的共同购买关系。
- 全局购物篮规则会掩盖不同价值客群之间的偏好差异。
- 数据清洗、指标计算和算法分析分散在不同脚本中，结果难以复用与追溯。
- 算法结果只有支持度和置信度，缺少面向经营人员的营销解释。

### 2.2 需要解决的核心问题

本项目聚焦以下四个问题：

1. 如何把百万级零售明细加工为口径统一、可复用的分层数据仓库？
2. 如何使用最近购买时间、购买频率和消费金额衡量客户价值？
3. 如何在可靠订单边界上挖掘商品组合，并识别不同客群的差异？
4. 如何把客户分层和关联规则转化为可查询、可解释、可展示的营销建议？

### 2.3 目标用户

| 用户角色 | 主要诉求 | 系统提供的能力 |
| --- | --- | --- |
| 经营管理者 | 快速了解销售与客户结构 | 经营总览、趋势、客群贡献率 |
| 用户运营人员 | 找到需要保持、发展或挽留的客户 | RFM 分层、客户列表、层级迁移 |
| 商品运营人员 | 设计组合促销和交叉销售 | 分群购物篮、关联网络、组合建议 |
| 数据分析人员 | 调整口径、阈值并验证结果 | 数据质量、规则筛选、指标明细 |
| 教师与答辩评委 | 验证大数据链路与项目完整性 | 数仓分层、算法指标、系统演示 |

## 3. 数据源与可行性

### 3.1 数据集说明

主数据集采用 UCI Machine Learning Repository 发布的 Online Retail II。它记录英国一家非门店在线零售企业从 2009 年 12 月到 2011 年 12 月的交易明细，许可为 CC BY 4.0。

- 官方页面：https://archive.ics.uci.edu/dataset/502/online+retail+ii
- DOI：https://doi.org/10.24432/C5CG6D
- 本地文件：`E:\bigdata\online+retail+ii\online_retail_II.xlsx`

原始文件包含两个年度工作表，字段一致：

| 原字段 | 业务含义 | 项目标准字段 | 主要用途 |
| --- | --- | --- | --- |
| Invoice | 发票号/订单号 | `invoice_no` | 购物篮边界、订单计数、取消识别 |
| StockCode | 商品编码 | `stock_code` | 商品标识、关联规则 |
| Description | 商品名称 | `description` | 展示与营销解释 |
| Quantity | 商品数量 | `quantity` | 销量、异常与取消判断 |
| InvoiceDate | 下单时间 | `invoice_ts` | Recency、趋势、动态快照 |
| Price | 商品单价 | `unit_price` | Monetary、销售额 |
| Customer ID | 客户编号 | `customer_id` | 客户粒度 RFM 与分群 |
| Country | 客户所在国家 | `country` | 地域维度分析 |

### 3.3 为什么适合本项目

该数据集同时具备标准 RFM 与购物篮分析所需的关键条件：

- `Customer ID` 提供稳定的客户边界。
- `Invoice` 明确标识一张真实订单，可直接定义一个购物篮。
- `InvoiceDate` 支持最近购买时间、复购间隔和按月快照。
- `Quantity × Price` 能计算标准 Monetary，而不需要用购买次数替代金额。
- `StockCode` 和 `Description` 支持频繁项集、关联规则和结果解释。
- 取消订单、负数量、缺失客户等问题能够体现 DWD 数据治理能力。

### 3.4 与原淘宝行为数据的关系

原淘宝数据拥有 12,256,906 条点击、收藏、加购和购买行为，适合漏斗和行为路径分析，但缺少订单号与价格，无法可靠界定购物篮，也无法计算标准 Monetary。

本项目第二阶段以 Online Retail II 作为唯一主交易数据源，保证订单与金额口径可靠。淘宝数据暂不并入主数仓，避免不同业务、时期和商品体系强行混合；如时间充足，可在扩展模块中作为行为漏斗案例展示，但不影响主项目验收。

## 6. 业务口径设计

### 6.1 购物篮边界

一张有效购物篮定义为：

> 同一个非取消 `invoice_no` 下，至少包含一个有效商品明细的集合。

同一商品在同一发票中出现多行时，先按商品编码汇总数量；购物篮关联挖掘只判断商品是否出现，不因购买数量重复计数。

### 6.2 有效交易明细

用于销售、RFM 和购物篮主分析的 DWD 明细满足：

- `invoice_no` 不以 `C` 开头。
- `quantity > 0`。
- `unit_price > 0`。
- `customer_id` 非空。
- `stock_code` 与 `invoice_ts` 非空。

被排除的明细不会直接丢失，而是保留在 ODS 和数据质量统计中。

### 6.3 RFM 定义

对于分析截止日 `snapshot_date`：

- `R`：截止日与客户最近一次有效购买日期的间隔天数，越小越好。
- `F`：观察窗口内不同有效发票号数量，越大越好。
- `M`：观察窗口内 `quantity × unit_price` 的有效消费金额之和，越大越好。

评分默认使用五分位数分箱，并保留分位点配置。客户分群基于 R、F、M 综合评分，初步设计为重要价值、重要保持、重要发展、重要挽留、潜力、新客户、一般和流失风险等类别。第二阶段将通过实际分布校验并固化规则。

### 6.4 关联规则指标

- 支持度：同时包含前件与后件的购物篮占全部目标购物篮的比例。
- 置信度：包含前件的购物篮中，同时包含后件的比例。
- 提升度：置信度相对后件自然出现概率的提升倍数。
- 覆盖购物篮数：规则共同出现的实际订单数量。

展示规则默认要求 `lift > 1`，并设置最低覆盖购物篮数，避免少量偶然共同购买造成误导。

## 7. 核心功能设计

### 7.1 经营分析总览

- 客户数、订单数、销售额、客单价、复购率。
- 月度销售额与订单趋势。
- 国家、商品和时段分析。
- 客群规模与销售贡献。
- 取消、缺失和异常数据概况。

### 7.2 动态 RFM 客户分层

- 选择快照月份和观察窗口。
- R/F/M 指标与评分查询。
- 客户价值矩阵和三维散点图。
- 客群规模、销售贡献与客单价对比。
- 相邻月份客群迁移桑基图。

### 7.3 分群购物篮分析

- 全局规则与指定 RFM 客群规则切换。
- 支持度、置信度、提升度和覆盖量筛选。
- 关联规则表、散点图、热力图和网络图。
- 同一商品在不同客群中的关联差异对比。

### 7.4 营销决策

- 输入目标客群与主推商品。
- 返回推荐搭配商品和规则依据。
- 展示历史覆盖订单、客户和销售金额。
- 输出组合折扣、交叉销售或客户挽留建议模板。

### 7.5 数据仓库与质量监控

- ODS、DWD、DWS、ADS 表和行数。
- 清洗前后数量核对。
- 质量规则、异常类型与处理结果。
- 数据加工链路和批次状态。

## 8. 创新点

### 8.1 从全局规则升级为分群规则

传统购物篮分析把所有订单混在一起。本项目先进行 RFM 分层，再对每个客群分别挖掘规则，使组合推荐同时考虑客户价值与商品偏好。

### 8.2 动态 RFM 与客群迁移

不是只在数据末日计算一次静态 RFM，而是按月形成快照，展示客户从新客户、潜力客户向重要价值客户发展，或向流失风险客户迁移的过程。

### 8.3 关联规则漂移分析

比较不同月份或年度的支持度、置信度和提升度变化，识别稳定组合、季节性组合和关系衰减，避免把一次性热点当成长期策略。

### 8.4 规则价值化排序

除算法指标外，增加覆盖订单数、覆盖客户数和历史销售金额，形成“统计显著性 + 业务价值”的综合排序。

### 8.5 数据质量可解释

将取消订单、负数量、零价格和客户缺失作为可展示的数据治理成果，明确哪些数据用于销售、RFM和购物篮，避免口径黑箱。

## 11. 技术栈总结

### 11.1 数据基础设施

| 技术 | 作用 | 选择理由 |
| --- | --- | --- |
| Docker Compose | 统一部署实验环境 | 工作区已有成熟配置，便于复现与答辩 |
| HDFS 2.7.4 | 原始与加工数据分布式存储 | 符合课程大数据平台要求，现有 3 DataNode 环境可复用 |
| Hive 2.3.2 | 数仓建模与离线 SQL 计算 | 已验证 ODS/DWD/DWS、分区和 MySQL 对接能力 |
| YARN / MapReduce | Hive 作业资源调度与执行 | 与现有 Hadoop 环境一致，无需引入新集群 |
| MySQL 8 | ADS 结果服务库 | 适合低延迟接口查询，避免前端直接访问 Hive |

### 11.2 数据处理与算法

| 技术 | 作用 | 选择理由 |
| --- | --- | --- |
| Python 3 | 数据转换、任务编排和算法 | 生态成熟，适合 RFM、关联规则与测试 |
| pandas / openpyxl | Excel 读取、CSV 转换与小规模核验 | 仅用于接入和校验，核心明细计算放在 Hive |
| Hive SQL | 清洗、维度建模、聚合与快照 | 计算口径清晰、可追溯、便于答辩展示 |
| mlxtend FP-Growth | 频繁项集与关联规则 | 比 Apriori 更适合较多购物篮，指标可解释 |
| NumPy / SciPy 稀疏结构 | 购物篮矩阵计算 | 降低商品布尔矩阵内存占用 |

算法阶段将按客群、月份和高频商品范围进行剪枝，并设置最低覆盖订单数，避免百万级明细直接构造无边界稠密矩阵。

## 12. 技术选型取舍

### 12.1 为什么不使用淘宝数据作为主数据

淘宝行为数据缺少订单号和价格。使用“同一用户同一小时”模拟购物篮会引入不可验证假设，购买次数代替 Monetary 也不符合标准 RFM。本项目优先保证口径可靠。

### 12.2 为什么不使用 Instacart 作为主数据

Instacart 的订单边界和复购结构很好，但公开竞赛数据缺少商品价格，无法直接计算标准 Monetary。Online Retail II 同时满足客户、订单、商品、时间和价格五项要求。

### 12.3 为什么当前不引入 Spark

数据规模约 106 万行，现有 HDFS/Hive/YARN 环境已经足以展示分布式存储、数仓和离线计算。额外建设 Spark 集群会增加环境风险，却不会显著改善项目得分或交付质量。

### 12.4 为什么使用 FP-Growth 而不是只用 Apriori

Apriori 会反复生成候选项集，商品较多时开销较大。FP-Growth 更适合本项目的购物篮规模；Apriori 可以作为小样本正确性对照，不作为主计算引擎。

### 12.5 为什么结果落 MySQL

Hive 适合批量计算，不适合前端频繁小查询。将 ADS 结果同步到 MySQL，既保留数仓计算链路，又能让 FastAPI 快速返回图表所需数据。

## 13. 风险与应对

| 风险 | 影响 | 应对方案 |
| --- | --- | --- |
| Customer ID 缺失较多 | 无法进入客户级 RFM | ODS 保留，质量页统计；从 RFM 主样本排除，不伪造客户 |
| 取消、退货和负数量 | 销售额与频次失真 | DWD 明确取消与异常标志，主分析和质量分析分开 |
| 商品数多、矩阵稀疏 | 关联规则内存或耗时过高 | 高频商品剪枝、客群分批、稀疏矩阵、最小覆盖阈值 |
| 部分商品名称缺失或不规范 | 规则展示难理解 | 以 StockCode 为主键，选择稳定名称并保留未知标记 |
| 英国数据占比过高 | 地域比较不均衡 | 总览如实展示，国家规则设置最低样本量，不夸大结论 |
| Hive 动态分区和小文件 | 运行速度不稳定 | 使用合理月分区、批量装载和中间汇总表 |
| 算法规则过多 | 前端难展示、业务不可用 | 综合支持度、提升度、覆盖量和销售额排序，限制 Top N |

## 14. 第二阶段编码边界与推荐顺序

第一阶段完成后，第二阶段按以下顺序实施：

1. 建立正式目录、配置模板和环境自检脚本。
2. 将两个 Excel 工作表转换为标准 CSV，输出接入核对报告。
3. 启动并验收 Hadoop、Hive、YARN 和 MySQL。
4. 编写并执行 ODS、DWD、DIM、DWS、ADS SQL。
5. 完成数据质量报告与分层一致性校验。
6. 实现静态和按月动态 RFM，验证分箱与分群分布。
7. 实现全局、分客群和分月份 FP-Growth 关联规则。
8. 将分析结果同步 MySQL，建立查询索引。
9. 实现 FastAPI 接口与自动化测试。
10. 实现 Vue/ECharts 五个核心页面并完成联调。
11. 编写一键启动、初始化和演示脚本。

第二阶段的首个可运行里程碑应为：

> 原始数据已入 HDFS，Hive 四层数仓可查询，能够输出一张可信的客户 RFM 结果表和一张有效购物篮汇总表。

## 15. 第一阶段结论

本项目的数据源、订单边界、金额字段、客户标识和时间跨度均满足标准 RFM 与购物篮分析要求；现有工作区已经具备 HDFS、Hive、YARN、MySQL、Python 和 Docker 基础环境，可以在不额外建设复杂集群的情况下完成完整大数据链路。

最终确定的技术路线是：

```text
HDFS + Hive 分层数仓
        ↓
动态 RFM 客户价值分层
        ↓
分客群 FP-Growth 购物篮挖掘
        ↓
MySQL + FastAPI 数据服务
        ↓
Vue + ECharts 精准营销决策展示
```

该路线兼顾技术完整性、业务价值、创新性、实现难度和答辩可展示性，可以进入第二阶段编码。
